# Анализ архитектуры и кодовой базы Street Fighter Card Game

В данном отчете представлен глубокий технический анализ клиентской карточной игры. Исследование проведено на основе исходного кода (монолит `game.js`, `index.html`, `style.css`), с фокусом на уязвимости, критические баги, баланс и архитектурные решения.

---

## 1. 🛑 CRITICAL BUGS

### 1. Бесконечное стаканье здоровья (Honda Passive)
- **Строки:** `512`, `696`, `774` (`startBattle`, `startArcadeBattle` и др.)
- **Описание:** Пассивный навык Хонды перед каждым боем выполняет `Game.player.maxHp += 15; Game.player.hp += 15;`. В режиме карьеры глобальный объект `Game.player` не сбрасывается между боями. Это означает, что к 8-му уровню базовое здоровье игрока мутирует и улетает в космос (+120 HP за забег).
- **Severity:** P0 (Критический баг, ломающий баланс)
- **Решение:** 
  Не изменять `maxHp` перманентно, либо ввести разделение на `baseMaxHp` и текущий модификатор.
  ```javascript
  // Решение 1 (если это отхил):
  if(Game.player.id === 'honda') { 
      Game.player.hp = Math.min(Game.player.maxHp, Game.player.hp + 15); 
  }
  // Решение 2 (если это бафф на начало боя):
  // Добавить свойство baseMaxHp в базу персонажей и применять только к нему.
  ```

### 2. Падение игры при битом `localStorage`
- **Строки:** `266`, `434`
- **Описание:** Вызов `JSON.parse(localStorage.getItem('sfStats'))` выполняется на верхнем уровне файла. В случае если в сторадже окажется поврежденный JSON, скрипт выбросит `SyntaxError`, выполнение `game.js` прервется и игрок получит белый экран/сломанную страницу без возможности восстановить сессию.
- **Severity:** P0
- **Решение:** Обернуть парсинг в безопасный try-catch:
  ```javascript
  let Stats = { wins: 0, losses: 0, /* ... defaults ... */ };
  try {
      const stored = localStorage.getItem('sfStats');
      if (stored) Stats = Object.assign(Stats, JSON.parse(stored));
  } catch(e) {
      console.error('Save data corrupted, using defaults');
  }
  ```

### 3. Неработающая пассивная защита (`def`)
- **Строки:** `1118 - 1130` (`dealDamage()`)
- **Описание:** Код вычисляет урон после применения плоской защиты: `let dmgAfterFlat = Math.max(0, amount - flatBlock);`. Однако дальше переменная `dmgAfterFlat` **вообще не используется** при финальном расчете. `finalDmg` считается как `amount - blocked`, полностью игнорируя параметр `def` у персонажей (например, у Zangief или Guile).
- **Severity:** P1 (Геймплейный баг)
- **Решение:** Заменить базу для вычислений:
  ```javascript
  let finalDmg = dmgAfterFlat - blocked; 
  ```

### 4. Эксплойт Супер-ударов (Отсутствие валидации EX)
- **Строки:** `884 - 920` (`playCard()`)
- **Описание:** Блокировка розыгрыша супер-карт (когда нет 100% EX) реализована исключительно в UI. Функция `playCard(index)` проверяет только затраты `pEnergy`, но не проверяет условие `reqEx`. Если вызвать `playCard` через консоль браузера, супер-удар сработает за 0 энергии и без EX шкалы.
- **Severity:** P2 (Игровой эксплойт)
- **Решение:** Добавить гвардию:
  ```javascript
  if(card.type === 'super' && Game.pEx < (card.reqEx || 100)) {
      logMsg("Недостаточно EX для супер-удара!", "red");
      return;
  }
  ```

### 5. Перманентная потеря одноразовых карт (`second_wind`)
- **Строки:** `175`, `906`
- **Описание:** Карта "Второе дыхание" имеет флаг `once: true`. При розыгрыше, из-за проверки `if(!card.once) Game.pDiscard.push(card);`, карта не попадает в стопку сброса. Перед следующим боем `startBattle()` собирает колоду только из руки и сброса. Из-за этого `once`-карта удаляется навсегда на весь оставшийся забег.
- **Severity:** P1
- **Решение:** Добавить зону "Exhaust" (сжигание).
  ```javascript
  if(!card.once) Game.pDiscard.push(card);
  else Game.pExhaust.push(card); // <- Новый массив
  
  // В startBattle():
  if(Game.pExhaust && Game.pExhaust.length > 0) { Game.pDeck.push(...Game.pExhaust); Game.pExhaust = []; }
  ```

### 6. Опровержение: "Deck Bloating" в `generateStartingDeck`
- **Опасение:** При вызове `startBattle()` происходит раздувание колоды?
- **Вердикт:** **Опасение ложно.** `generateStartingDeck()` возвращает новый массив только при вызове `startCareer()`. `startBattle()` (строки 751-755) лишь перекладывает карты из `pHand` и `pDiscard` обратно в массив `pDeck`. Это корректная логика восстановления Draw Pile перед боем, дубликатов не создается.

## 2. ⚖️ GAME BALANCE

- **Математика Сложности (Difficulty Scaling):** Таблица `DIFFICULTY` на сложности "Мастер" (уровень 2) применяет мультипликативные штрафы: Враги получают 160% HP и 200% Урона, а игрок получает 40% Золота, при этом цены в магазине 160%. Это двойное наказание (И враги сильнее, И игрок беднее), которое делает "Мастера" зависимым исключительно от рандома. Рекомендуется балансировать либо экономику, либо боевые статы, но не оба аспекта сразу с такими агрессивными коэффициентами.
- **Баланс Akuma:** Описание (строка 153) гласит: *"Каждый 3-й бой +1 к атаке"*. В коде (`dealDamage`, строка 962) реализован только перманентный бонус к урону (`rawDmg * 1.5`). Механика прогрессии атаки каждые 3 боя отсутствует. Либо необходимо исправить описание, либо дописать логику в `startBattle()`.
- **Защитные персонажи:** Из-за бага с `dmgAfterFlat` (см. пункт 1) персонажи, полагающиеся на `def`, на данный момент неиграбельны на высоких уровнях карьеры.

## 3. 🔧 CODE ARCHITECTURE

- **Проблема монолита:** `game.js` (1750+ строк, 89KB) содержит бизнес-логику, UI-рендеринг, базу карт и настройки Particle-эффектов. Дальнейшее масштабирование (например, добавление новых фаз или PvP) превратится в спагетти-код.
- **Менеджмент состояния (`Game` Singleton):** Смешивание Session-state (колода карьеры, уровень, золото) и Battle-state (баффы, рука, текущее HP) в одном глобальном объекте усложняет реализацию сохранений и приводит к багам вроде бага Хонды.
- **Card Immutability:** Колоды собираются через shallow copy: `{...card, uid: Math.random()}`. Использование `Math.random()` в качестве UID делает игру не-детерминированной (невозможно будет добавить seed для "Daily Runs").

**Рекомендация (Long-term):** Перевести проект на ES-модули: `state.js`, `cards.js`, `battle.js`, `ui.js`.

## 4. 🎮 UX / UI

- **Input Lock:** Отсутствует блокировка ввода во время выполнения анимаций/ходов врага. Игрок может сделать "double click" по карте, что вызовет непредсказуемое поведение UI и стейт-машины.
- **Battle Log:** Бесконечно растёт в течение боя (`document.getElementById('battle-log').innerHTML += ...`), потребляя DOM узлы. Необходимо ограничить лог последними 10-20 сообщениями.
- **HP Bar:** Отсутствует визуальный фидбек критического уровня здоровья (Цвет всегда одинаковый). Добавление смены цвета (Зеленый -> Желтый -> Красный) сильно улучшит UX.

## 5. 🚀 PERFORMANCE

- **Бесконечный `requestAnimationFrame`:**
  В модуле `FX` (строки 40-75) метод `loop()` запускается при инициализации игры и крутится бесконечно:
  ```javascript
  requestAnimationFrame(() => this.loop());
  ```
  Это заставляет браузер стирать холст 60 раз в секунду, даже когда игрок находится в главном меню или читает диалоги. **Решение:** останавливать цикл, когда `this.particles.length === 0` и заново вызывать `requestAnimationFrame` внутри `FX.emit()`.
- **DOM Rebuild:** Метод `updateUI()` каждый раз с нуля перестраивает HTML (через `.innerHTML = ...`) для карт в руке, статусов и баффов. При добавлении тяжелой анимации это будет вызывать подтормаживания (Jank).

## 6. 🔒 SECURITY

- **XSS-уязвимость в `renderAvatar`:**
  Строки 103-106 возвращают строку прямо в DOM через `.innerHTML`.
  ```javascript
  if (avatar.startsWith('data:image')) return '<img src="' + avatar + '">';
  return avatar; // <-- RAW HTML
  ```
  Если в будущем аватары будут браться из JSON-сохранений или с сервера, это прямая инъекция `<script>`. Необходимо использовать `document.createElement` или хотя бы санитайзер.
- **Client-side Math.random:** Использование стандартного рандома означает, что клиент имеет полный контроль над исходами.
- **Save Manipulation:** Сохранения (статистика) лежат в `localStorage` в открытом виде.
